# 功能说明书 (Functional Spec)
项目的名称为“个人智能幕僚”，代号Staff。

## 1. 核心模块定义 (待评估与选型)

### 1.1 接入信道层 (Multi-Channel Plug-and-Play Gateway)
- **定位**：对内处理标准的数据流转结构，对外通过轻量级网络协议对接多元化通信基础设施。
- **插座模式 (Provider Logic)**：全面重构通信层架构，将微信等通信服务视作类似于大模型（LLM Providers）的**“插头”**。系统提供标准的“插座接口”，允许通过配置动态挂接、卸载、管理多个异构的微信服务实例（如 Gewechat-Instance-A, Gewechat-Instance-B）或其他社交平台服务。
- **多实例支持与路由**：每个挂接的通信实例拥有独立的配置（API Base、Token、鉴权策略、白名单），系统根据接收到的 Event Source 进行自动路由和鉴权分发。
- **多模态扩展**：在此对接点可拦截语音与图片，调用外置大模型提前提取纯文字描述后一并向后传递。

### 1.2 主客体分发与隔离引擎 (Routing & Sandboxing)
- **主人全能模式 (Master Mode)**：系统管理员视角，拥有系统底层配置权限与全组资源读取权限角色。
- **外部来访者模式 (Guest/Staff Mode)**：为每个外部对接者或群组提供彼此之间绝对的会话上下文隔离区。

### 1.3 业务控制逻辑组件 (Skills Enhancement)
在未来的编排引擎中：
- **【边界墙】权限拦截模块**：通过下发限定系统指令，使得运行态的助手理解对外职能限制，识别管理与配置等越权询问并予以婉拒。
- **【安抚机】异步工单跟进**：设计一套独立于问答之外的任务追踪环，在产生请示型异步挂起时创建监听工单，超时未被处理即下发关怀类状态回复。
- **【降噪层】上下文治理 (Context Optimization)**：【新增】引入活跃会话窗口机制与历史文本裁剪。自动拦截非当前论次的冗余 `tool_result`，防止上下文膨胀导致的回复质量下降。
- **【跨会话】力所能及的协助**：【新增】允许 Staff 在访客模式下使用 `search_contacts` 和 `send_cross_chat` 工具，主动帮助访客完成非敏感的通知与联系任务。
- **【画像搜集】自动化身份补全与提纯**：【持续优化】
  - **画像缺失驱动 (Pillar Discovery)**：系统不再仅基于“新访客”判定，而是常态化扫描访客记忆中的占位符（如 `未知访客`、`暂无别名`、`(待收集)`）。只要核心字段缺失，即激活引导 SOP。
  - **确定性同步 (Deterministic Sync) [NEW]**：
    - **同步策略**：系统不再依赖 LLM 的“梦境提纯”来获取钉钉原生字段（如姓名、部门、职位、组织路径）。
    - **原子化段落重绘 (Section-based Atomic Rewrite)**：
      - **锚点驱动**：回写引擎锁定 `### 🎭 基本特质与履历` 锚点，提取并全量刷新该段落。
      - **锁行格式**：强制执行 `标签: 新值` 格式，物理覆盖并清理之前因同步异常产生的“孤立行”数据，确保 MD 列表结构的语义完整性。
    - **更新频率**：执行“每日首次接触同步”策略。系统记录 `last_sync_date`，若当日首次发生对话，则强制触发一次完整的 API 资料拉取与回填。
  - **深度接口补全 (Deep Fetching)**：
    - **优先级**：首选旧版 OAPI (`/user/get`) 以确保稳定性，新版 V1.0 作为兜底。
    - **字段扩展**：除基础字段外，需抓取电子邮箱 (`email`)、直属领导姓名 (`manager_name`)。
    - **部门翻译引擎 [OPTIMIZED]**：
      - **字段对齐**：由于 DingTalk v2.0 内部差异，系统策略为：优先读取 `result.name` 字段，其次 `dept_name` 作为兜底，彻底解决翻译返回 ID 的问题。
      - **动态根解析 (Dynamic Root)**：不预置根部门（ID: 1）名称，由 API 实时请求，确保企业命名真实性。
      - **内存翻译缓存**：引入 `_dept_names` 内存映射，防止多路径解析时的 API 限频触发。
      - **文案纠偏**：全局拦截“全公司”文案，统一将其转译为更专业的“总公司”。
    - **组织架构全路径 [NEW]**：需调用 `/topapi/v2/department/listparentbyuser` 获取用户所属的所有父级部门，并解析为直观的组织架构路径。
    - **递归解析**：对于直属领导，系统应自动执行二次查询以获取其姓名，而非仅存储 ID。
  - **资源优化控制 (Resource Guard)**：为防止异步提纯前的重复调用，系统引入内存标记位。在单次服务运行周期内，针对同一用户仅触发一次深度查询。
  - **梦境提纯锚点**：系统需重点提纯 `Identity` (身份/背景/职位/邮箱), `Role & Projects` (关注项目), `Relationship` (与主人的关系/直属领导)。提纯应由 Metadata Hint 驱动。
- **【闭环机制】工单反馈通知 (Ticket Feedback Notification)**：当主人（Master）对工单执行批准（Approve）或驳回（Reject）操作时，系统必须自动向工单发起者发送处理结果。通知需包含：
  - **身份标识**：消息头部必须注入工单 ID（格式如 `✅ **【工单回复】 {ticket_id}**`），确保访客在多工单并行时能准确对应。
  - **内容要素**：包含工单摘要、处理结果、具体理由及后续引导。

- **【静默模式】降噪守护 (Silent Guard)**：【新增】
  - 引入 `[SILENT]` 标记分级。允许低价值或常规性汇报仅通过 `Monitor` 网页前端展示（影子日志），不再强推钉钉，实现非侵入式后台运行。
  - 通过 `heartbeat.jsonl` 作为持久化中心，兼容 Guest 引用机制下的上下文还原。

## 2. API 与逻辑数据流概览
标准的外部消息处理流水线：
`外部消息 -> Gateway(多模态提取) -> Router(身份判定) -> Sanitizer(净化指令) -> Brain(意图分析) -> RAG/外部API调用 -> Output Audit(输出审计防脱敏) -> 向外推送响应`
