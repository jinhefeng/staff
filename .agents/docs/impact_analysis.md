## 3. 上下游依赖及潜在影响评估 (预警项)

本次开发计划引入的新功能和架构模块，其最大的影响和风险来自于外部环境依赖，具体预警如下：

### 🚨 预警一：微信风控与 Hook 稳定性 (接入层)
- **依赖模块**：微信机器人底层框架（基于 iPad/Mac 协议）。
- **影响分析**：
  - 微信随时可能进行协议升级或封包检测，这会导致 Staff 在不可抗力下离线。
  - **缓解策略**：在“自动安抚机”与“异步闭环”之外，必须引入**“脱机自救机制”（心跳检测与告警）**。当检测到微信断线时，通过备用通道（如企微、邮件或短信）立刻向 Master (主人) 报警。

### 🚨 预警二：Prompt Injection (注入攻击) 突破 Sandbox (安全层)
- **依赖模块**：Sanitizer Sandbox (Prompt 净化层)。
- **影响分析**：
  - “越狱”手段层出不穷，单靠正则或前置 LLM 审查可能存在漏网之鱼。如果外部攻击成功诱导 Staff 吐出 Master 的 Private Rules，将导致严重隐私泄露。
  - **缓解策略**：物理层面限制。即使是被攻破的 Staff Engine，其运行时上下文中也**绝对不应该加载**高度敏感的 Private Rules（仅加载可对外公开的 SOP），真正实现内存级别的安全物理隔离。

### 🚨 预警三：大语言模型 API 延迟毛刺 (大脑中枢)
- **依赖模块**：OpenClawd大模型中枢、意图识别、RAG。
- **影响分析**：
  - 高峰期由于 API 限流或延迟，导致“回复”时间超长。在此期间自动安抚机（Auto-Pacifier）如果缺乏锁机制，可能会并发多次触发安抚。
  - **缓解策略**：为每个外部请求建立全局唯一的 Ticket ID（工单化管理），利用 Redis 或本地 SQLite 记录状态机，防止同一个 intent 触发多次安抚下发。

## 4. 实施阶段建议
基于上述分析，本架构方案在技术上完全可行，但需要在开发“接入层”和“记忆系统”时，引入严密的异常捕捉和状态机管理体系。建议首先进行“中枢引擎（Brain）”和“双内核隔离”的基础骨架搭建，并用 mock 数据进行接口联调。
