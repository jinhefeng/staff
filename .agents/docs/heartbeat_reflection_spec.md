# Heartbeat 与梦境提纯消息机制说明 

## 背景
HeartbeatService 会定期唤醒 AI 幕僚检查后台工单（任务）或在闲暇时执行“梦境提纯”（潜意识反思，Subconscious Reflection）。

## 消息通知机制优化
为减少对用户的打扰，同时保证系统运行透明度，针对“梦境提纯”任务的通知机制做如下规定：
1. **梦境提纯开始时**：不再发送启始通知消息。系统在后台默默启动记忆处理过程。
2. **梦境提纯结束时**：完结时需发送消息，且包含提纯结果。将模型提纯的实际结果摘要作为汇报内容，直接呈现给用户（Master），使其知道提纯正在正确运行。
3. **普通异步工单任务**：保持原有的“开始启动”与“最终完结汇报”的双重通知逻辑不变。

## 提示动态同步 (Dynamic Instruction Sync)
为保证系统配置与说明文档的一致性，`HeartbeatService` 必须具备自动校对能力：
- **触发机制**：每次心跳周期（`_tick`）开始时。
- **动作内容**：读取 `config.json` 中的 `intervalS`，并使用正则表达式自动刷新 `HEARTBEAT.md` 头部关于轮询周期的文字描述（中英文）。
- **目的**：消除人工修改配置后文档滞后的风险。

## 归档一致性设计 (Archiving Consistency)
系统必须保证 Markdown 视图与 JSON 数据库的强原子性同步：
- **物理归档联动**：当 Heartbeat 标记任务为已完成（`[x]`）时，**必须**同步触发 `TicketManager.resolve_ticket(ticket_id)`。
- **双保险匹配**：标记逻辑应同时支持“任务全描述匹配”和“Ticket ID 锚点匹配”，两路路径最终都必须指向物理数据库的归档操作。
- **清理逻辑**：在每次心跳开始时，执行 `_archive_completed_tasks` 将已完成的 `[x]` 条目物理移动至 `## Completed` 区域，保持 `Active Tasks` 区域的整洁。
