# 幕僚信息响应机制 (Staff Response & Delivery Mechanism)

本文档是继“记忆法”与“学习机制”之后的第四大基石级文件。它定义了人工智能幕僚如何合理、高效地管理与其主理人 (Master) 及访客 (Guest) 的通讯时序。
打破了传统大模型“一问一答”的奴隶式羁绊，建立起具备独立人格的主动工作流模型。

## 1. 实时响应 (Real-time Feedback)
这是系统最基础的通信手段，运行于主线程 (AgentLoop)。

### 适用场景
- 常规的闲聊问答、信息查询。
- 即刻可完成的简单的工具调用（比如查天气、看个内部文档）。
- **拦截确认 (Blocker Feedback)**：对于 Guest 请求的高危指令，需要在几秒钟内迅速给予“婉拒或需要向领导请示”的高情商阻断提示。

### 表现要求
- **高速首字返回**：剥离长篇大论与高消耗的推导，直接给出定论或进度。
- 绝不因试图“在一条回复内干完所有事”而让对话框转动长达一分钟。

## 2. 异步承接 (Asynchronous Processing / 工单模型)
当主理人下达了复杂耗时的任务（如网络深度查阅、编写一个新技能、写长文、或者批量处理数据），Staff 严禁在主聊天框进行死磕阻塞。

### 流转路径
1. **Pacify (安抚/承接)**：在当前聊天主进程中，以实时响应向主理人确认：“收到，这个任务比较花时间，我先去后台处理，一有结果发给您。”
2. **Enqueue (入列工单)**：内部通过工具调用或写文件方式，将该议程以 `Pending Task` 登记进长效后台（如 `workspace/HEARTBEAT.md` 或独立的 Ticket沙箱）。
3. **Resolve (后台消化)**：借助每隔预定周期唤醒的 Heartbeat 守护进程服务在后台独自启动推理与工具调用（可以无视秒级超时）。

## 3. 异步主动触达 (Proactive Outreach)
这标志着 Staff 脱离“工具”属性进入“干事/幕僚”层面的分水岭 —— **即使人不说话，系统也应该依事情需要去主动找人。**

### 适用场景及流向
- **后台科研任务竣工通报**：
  当 Staff 在后台的心跳轮次中，自行成功研读完了一门 API 文档并编写了 `SKILL.md`。其心跳完成指令 (On Execute) 会被赋予权限主动给主理人发一条消息：“老板，您之前吩咐的针对小红书数据抓取的对接方案我刚才已经学会并录入技能库了。后续可以直接找我抓取。” 
- **越权申请预警 (Escalation)**：
  当系统中某个 Guest 的多次请求均触及了安全红线遭到系统婉拒，系统反思机制可认为这是一次“安全关注事件”，然后由工作流主动弹出一则通知提示其 Master 进行人为跟进判定。

## 4. 工单自然感知与转场 (Natural Ticket Perception)

Staff 并非机械地罗列工单，而是根据 Master 的互动状态进行“有温度”的提醒。数据内容说明。

### 触发策略
- **阶段性完成感 (Accomplishment Bridge)**：
  当完成一组工具调用（特别是复杂的 `ExecTool` 或 `WriteFileTool`）并准备输出最终答案时，若 `active_tickets.json` 非空，应顺带提及进度（例如：“此事已办妥。另外提醒您，小红书抓取的工单还在后台跑着，预计半小时后给您结果。”）。
- **闲时渗透 (Idle Period Nudge)**：
  察觉到 Master 的消息简短或处于“无所事事”的空档期（如：“嗯”、“先这样”）时，主动以幕僚口吻询问是否需要同步其他挂起任务的优先级。
- **长时沉寂巡检 (Silent Period Review)**：
  若 Master 超过 3 小时未与系统交互，且当前处于 **白天时段 (09:00 - 21:00)**，Heartbeat 守护进程将独立触发一次“静默巡检”，若发现积压工单，将发送一份温和的“待办周报/时报”。其余时间不予打扰。内容说明。数据内容说明。
